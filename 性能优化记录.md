# 星途导航性能优化记录

> 一个 267 张卡片、15 个分类的静态导航站，如何从滚动卡顿优化到丝滑流畅？

## 背景

星途导航（StarNav）是一个基于 Nuxt 4 + TypeScript + Element Plus + Tailwind CSS 的纯静态导航站，首页一次性渲染 **267 个网站卡片**。页面功能完善后，滚动时明显感到**卡顿掉帧**，尤其在快速滚动时尤为严重。

## 问题诊断

通过 Chrome DevTools Performance 面板分析，发现了三个核心瓶颈：

### 1. GPU 合成层爆炸

```css
/* 当时所有卡片上静态挂着这个样式 */
.hover\:will-change-transform {
  will-change: transform;
  transform: translateZ(0);
}
```

`will-change: transform` 会告诉浏览器提前为元素创建独立的 GPU 合成层。这本是一个好的性能提示——但问题在于：**267 张卡片全部静态持有这个属性**，意味着浏览器要为 267 个元素维护独立的 GPU 层。显存占用飙升，滚动时 GPU 需要管理 267 个层的合成与重排，**反而拖慢了性能**。

### 2. DOM 节点过多

`NavGrid.vue` 一次性渲染所有分类和所有卡片。首屏可能只需要看到 1-2 个分类（约 30-40 张卡片），但浏览器要为 **267 个卡片元素**全部参与布局计算、样式计算和渲染树构建。

即使图片还没加载（`loading="lazy"`），img 标签本身也是**活生生的 DOM 节点**——每次滚动、每次重排，浏览器都要重新计算所有 267 个卡片的位置和尺寸。

### 3. 并发网络请求拥堵

每个 `NavCard` 都需要加载一个外部 favicon：

```vue
<img :src="getIconUrl(item)" loading="lazy" decoding="async" />
```

`loading="lazy"` 虽然能延迟图片加载，但当用户滚动到一个分类区块（比如 AI 工具的 36 个网站）时，浏览器会**同时触发这个区域内所有图片开始加载**——36 个请求瞬间涌出，填满 HTTP 连接池，慢的 favicon 还会阻塞快的。

## 优化策略：三层递进懒加载

我们的核心思路是：**从外到内、层层递进地控制渲染粒度**，而不是一次性把所有东西都交给浏览器。

```
离视口很远 ──→ 只显示骨架屏（区块级）
离视口 300px ──→ 渲染卡片 div + 兜底图标（卡片级）
离视口 200px ──→ 才创建 img 标签加载 favicon（图片级）
```

### 第一层：区块级懒加载（InViewRender）

**解决的问题：** DOM 节点过多，267 张卡片全部参与布局计算。

**做法：** 创建一个通用的 `InViewRender` 组件，用 IntersectionObserver 控制每个分类区块的渲染时机：

```vue
<InViewRender :id="`category-${cat.id}`" :item-count="cat.items.length">
  <!-- 可见时才渲染这里的卡片内容 -->
  <NavCard v-for="item in cat.items" :key="item.id" :item="item" />
</InViewRender>
```

- 不可见时：只显示几行**骨架屏占位**（`animate-pulse`），不渲染任何 NavCard
- 进入视口前 300px：IntersectionObserver 触发，渲染真实内容
- 站长推荐分类（内容少）直接渲染，不懒加载

**效果：** 首屏 DOM 节点从 267 个减少到 30-50 个。

### 第二层：卡片级图片懒加载

**解决的问题：** 浏览器 `loading="lazy"` 仍然会导致同一区域大量图片并发请求。

**做法：** 在 `NavCard` 内部再加一层 IntersectionObserver，控制 img 标签的创建时机：

```vue
<div ref="cardRef">
  <!-- 未进入视口前只显示兜底图标 -->
  <div v-if="!shouldLoad || !iconLoaded">
    <Icon :icon="getFallbackIcon(item)" />
  </div>

  <!-- 进入视口前 200px 才创建 img 标签 -->
  <img v-if="shouldLoad && !iconError" :src="getIconUrl(item)" ... />
</div>
```

```ts
onMounted(() => {
  observer = new IntersectionObserver(([entry]) => {
    if (entry.isIntersecting) {
      shouldLoad.value = true  // 触发 img 标签渲染
      observer.disconnect()
    }
  }, { rootMargin: '200px' })
  observer.observe(cardRef.value)
})
```

**关键区别：** 这不是定时器驱动的"排队"，也不是手动控制的顺序加载。而是由**滚动行为自然驱动**——浏览器 Event Loop 处理 IntersectionObserver 回调，创建 img 标签，然后 HTTP 连接池自动管理并发数（同一域名通常 6 个连接）。当 img 标签数量从 267 降到只有视口附近的 15-20 个时，连接池压力自然消失。

### 第三层：will-change 动态化

**解决的问题：** 267 张卡片静态持有 GPU 合成层。

**做法：** 从静态属性改为 CSS 伪类动态触发：

```css
/* 之前：静态持有，267 个 GPU 层常驻 */
.hover\:will-change-transform {
  will-change: transform;
  transform: translateZ(0);
}

/* 之后：hover 时临时启用，释放即销毁 */
.group:hover {
  will-change: transform;
}
```

**效果：** GPU 合成层从 267 个常驻变为仅 hover 瞬间临时创建，释放后浏览器自动回收。

## 为什么浏览器自带的 loading="lazy" 不够？

这是很多人的疑问。关键在于**控制粒度**不同：

| 对比项 | 浏览器 `loading="lazy"` | 我们的 IntersectionObserver 方案 |
|--------|------------------------|-------------------------------|
| img 元素 | ✅ 一直存在于 DOM 中 | ❌ 不可见时连 img 标签都没有 |
| 加载时机 | 浏览器自己判断 | 开发者控制（提前 200px 预加载） |
| DOM 开销 | 所有卡片都有 img 元素参与布局 | 不可见的卡片只有轻量兜底图标 |
| 并发控制 | 不可控，一批几十个请求同时发起 | 自然排队，滚动到哪加载到哪 |

形象地说：

> `loading="lazy"` 像"让 267 个人排队等叫号，但所有人都在候诊室里站着"
> 
> 我们的方案像"只让快到号的人进候诊室，其他人都在家等通知"

前者 267 个人挤在候诊室（DOM 布局计算）→ 卡顿
后者候诊室只有十几个人 → 流畅

## 优化效果

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| 首屏 DOM 节点数 | ~267 个卡片 | ~30 个卡片（仅站长推荐） |
| GPU 合成层 | 267 个常驻 | 0 个常驻，hover 瞬态创建 |
| 同时发起的 HTTP 请求 | 数十个并发 | 视口附近 15-20 个 |
| 滚动帧率 | 明显卡顿掉帧 | 60fps 流畅 |

## 技术要点小结

1. **不要静态持有 `will-change`** — 它应该只在动画发生前短暂启用，用完让浏览器自动回收
2. **DOM 节点数量是第一大敌人** — 即使元素不可见，参与布局计算的节点太多也会导致滚动卡顿
3. **多层懒加载比单层更有效** — 区块级 → 卡片级 → 图片级，层层递进地控制渲染粒度
4. **IntersectionObserver 比 loading="lazy" 更可控** — 你可以精确控制提前量（rootMargin），让图片在到达视口前就加载完成
